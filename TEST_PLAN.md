# 主播全生命周期管理系统 — 测试计划

> 适用版本：当前主分支
> 编写日期：2026-05-24

本测试计划覆盖**功能用例 + 权限矩阵 + 端到端场景 + 边界与安全**四类，旨在一次完整回归。建议按顺序执行，前置数据后置依赖明确。

---

## 0. 测试准备

### 0.1 环境

- 操作系统：Windows 10/11
- Python：3.11（虚拟环境位于 `backend\.venv`）
- Node：18+（`frontend\node_modules` 已就绪）

### 0.2 启动

```powershell
# 后端（端口 8000）
cd D:\zhubo-life-assisstant\backend
.\.venv\Scripts\python.exe -m uvicorn app.main:app --reload --port 8000

# 前端（端口 5173）
cd D:\zhubo-life-assisstant\frontend
npm run dev
```

打开浏览器访问 `http://localhost:5173`。

### 0.3 默认账号

| 用户名 | 密码 | 角色 | 主要权限 |
|---|---|---|---|
| admin | admin123 | 管理员 | 全部 |
| operator | operator123 | 运营 | 业务读写，无薪资写、无合同终止、无用户管理 |
| finance | finance123 | 财务 | 薪资读写，业务只读 |

### 0.4 测试数据基线

每轮测试前建议手动清理 `backend/zhubo_sys.db`（停后端 → 删库 → 重启，会自动建表并种子三个账号），以保证用例编号 1 起的数据可重现。

> 提示：删库前如有正式数据请先备份。

## 1. 模块用例

每条用例格式：

- **前置**：执行前需达到的状态
- **步骤**：用户操作
- **预期**：可观察的结果（UI 行为 + 后端数据）
- **回归点**：验证完毕后系统应处于的状态

---

### 1.1 认证与会话（AUTH）

#### TC-AUTH-01 正常登录

- **前置**：未登录
- **步骤**：
  1. 访问 `/login`
  2. 输入 `admin / admin123`，点击"登录"
- **预期**：
  - 跳转到 `/`（主播列表）
  - 顶栏显示 admin 身份
  - DevTools → Application → Session Storage 存在 `zhubo_token` 与 `zhubo_user`
  - DevTools → Application → Local Storage **不应存在** 这两个键

#### TC-AUTH-02 错误密码

- **步骤**：登录页输入 `admin / wrong`
- **预期**：页面顶部红色 Alert 提示"用户名或密码错误"，未跳转

#### TC-AUTH-03 必填校验

- **步骤**：账号或密码留空点击"登录"
- **预期**：对应输入框下方提示"请输入用户名/密码"

#### TC-AUTH-04 关闭浏览器后失效（核心需求）

- **前置**：以 admin 登录后任意页面
- **步骤**：
  1. 关闭浏览器**所有**窗口
  2. 重新打开浏览器，访问 `http://localhost:5173`
- **预期**：自动跳转 `/login`，需重新登录

#### TC-AUTH-05 刷新页面保持登录

- **前置**：admin 登录后停留在 `/dashboard`
- **步骤**：按 F5 刷新
- **预期**：仍停留在 `/dashboard`，无需重新登录

#### TC-AUTH-06 401 自动跳登录

- **前置**：admin 登录后停留在主播列表
- **步骤**：
  1. DevTools → Application → Session Storage → 把 `zhubo_token` 改成乱码
  2. 在页面执行任意刷新动作（点搜索/筛选）
- **预期**：自动跳转 `/login`，session storage 中 token/user 被清空

#### TC-AUTH-07 修改自己密码

- **前置**：以 operator 登录
- **步骤**：右上角头像 → 修改密码 → 输入旧密码 `operator123`，新密码 `Op@123456`，确认
- **预期**：
  - 提示"修改成功"
  - 退出登录，用旧密码登录失败，新密码登录成功
- **回归点**：把密码改回 `operator123`，避免影响后续用例

#### TC-AUTH-08 登录页不出现演示账号

- **预期**：登录页除标题、表单、登录按钮外，**没有**"演示账号（点击自动填充）"区块（需求项）

#### TC-ANC-02 创建主播（全字段）

- **步骤**：填写电话、身份证、平台、银行卡、紧急联系人、风格标签等
- **预期**：保存成功；详情页能看到所有字段

- **步骤**：
  1. 在搜索框输入"小三"
  2. 状态切换到"实习"
- **预期**：仅显示匹配主播

#### TC-ANC-04 编辑主播

- **步骤**：进入详情 → 编辑 → 修改电话
- **预期**：返回详情页电话已更新；操作日志出现一条 update 记录

#### TC-ANC-05 删除主播 → 材料库清理（核心需求）

- **前置**：该主播已在节点上传过文件
- **步骤**：在详情页点击"删除主播"
- **预期**：
  - 主播从列表消失
  - **材料库 4 个分类下，该主播 ID 的目录全部消失**
  - 磁盘上 `backend/files/contracts/<id>` 等目录已被删除
  - 操作日志出现 delete 记录

#### TC-ANC-06 成长阶段计算

- **步骤**：创建 4 个主播，入职日期分别设为 `今天`、`100 天前`、`200 天前`、`400 天前`
- **预期**：列表显示阶段分别为：新人期、成长期、成熟期、头部期

#### TC-ANC-07 直播统计聚合（来自后端 batch 接口）

- **前置**：在某主播下创建多条直播记录（GMV/观看人数不同）
- **步骤**：返回主播列表
- **预期**：该主播卡片上显示场次、累计 GMV，与详情页"直播数据" Tab 一致

### 1.5 材料库（LIB）

#### TC-LIB-01 目录树展示

- **前置**：节点已上传若干文件
- **步骤**：访问"材料库"
- **预期**：4 个分类节点 → 主播艺名子节点 → 文件节点

#### TC-LIB-02 文件预览

- **步骤**：点击文件节点
- **预期**：浏览器新标签预览（PDF / 图片）

#### TC-LIB-03 单文件删除

- **步骤**：点击文件右侧红色删除按钮 → 确认
- **预期**：
  - 文件从树中消失
  - 磁盘对应文件已删除
  - DB 中 `FileRecord` 记录被清

#### TC-LIB-04 主播目录删除

- **步骤**：点击主播子目录右侧删除按钮 → 确认
- **预期**：该主播在该分类下所有文件被一并删除（磁盘 + DB）

#### TC-LIB-05 一键清理孤儿（核心需求）

- **前置**：DB 中无该主播但磁盘上仍有目录（可手工删一个主播或本机已存在残留）
- **步骤**：
  1. 进入材料库，孤儿目录显示红色"孤儿"标签
  2. 点击右上角"清理孤儿目录" → 确认
- **预期**：
  - 提示已清理 N 个目录
  - 树中孤儿目录消失
  - 按钮再次变灰（无孤儿）

#### TC-LIB-06 路径穿越防护（安全）

- **步骤**：手工调用
  ```
  curl -X DELETE "http://localhost:8000/api/library/folder?path=/contracts/../../etc" -H "Authorization: Bearer <admin token>"
  ```
- **预期**：返回 400 "无效路径"；磁盘上未发生变化

#### TC-LIB-07 不可删分类根目录

- **步骤**：调用 `DELETE /api/library/folder?path=/contracts`
- **预期**：返回 400 "不允许删除分类根目录"

## 3. 端到端场景（E2E）

#### TC-E2E-01 新人入职到首场直播

1. admin 创建主播 `李四`
2. 添加 `入职` 节点并上传合同 PDF
3. 创建合同（正式 / 1 年）
4. 创建并绑定直播间、借出设备
5. 录入第一场直播记录
6. 在材料库与看板上验证可见
- **预期**：每一步都成功；看板上 `李四` 阶段为新人期、合同生效中、设备借出中、直播 1 场

#### TC-E2E-02 培训 → 直播效果回归

1. 创建培训计划（线下）
2. 主播报名并签到
3. 培训前 1 天 + 培训后 3 天分别录入直播记录
4. 在考勤行查看"效果分析"
- **预期**：返回培训前后窗口的 GMV/观看/订单差值

#### TC-E2E-03 月度结算

1. 财务设置默认配置（底薪 5000，提成 5%）
2. 选某主播录入若干直播 GMV 共计 10w
3. 预览 → 生成 → 发放
- **预期**：薪资=底薪+提成（按规则）；操作日志全链路可追溯

#### TC-E2E-04 离职归档

1. operator 给主播提交离职
2. admin 推进到"已完成"
3. 主播自动变 `离职`
4. 删除主播（admin）
- **预期**：主播消失；磁盘材料库目录被同步清理；看板主播总数 -1

#### TC-E2E-05 孤儿目录修复（核心需求）

1. 在材料库可看到截图中的孤儿目录
2. 点击"清理孤儿目录" → 确认
3. 刷新材料库
- **预期**：孤儿目录全部消失，按钮变灰

## 4. 边界与安全（SEC）

#### TC-SEC-01 长字符串

- **步骤**：主播姓名输入 256 个字 + 表情符号
- **预期**：要么截断保存、要么前端校验提示，不能 500

#### TC-SEC-02 文件大小限制

- **步骤**：上传 60MB 文件
- **预期**：返回"文件大小超过限制"

#### TC-SEC-03 文件扩展名白名单

- **步骤**：上传 `a.exe`、`a.zip`
- **预期**：返回"不支持的文件类型"

#### TC-SEC-04 路径穿越（材料库 API）

- 见 TC-LIB-06

#### TC-SEC-05 SQL 注入

- **步骤**：在主播搜索框输入 `' OR '1'='1`
- **预期**：返回 0 条或精确匹配，不报错（SQLAlchemy 参数化已防护）

#### TC-SEC-06 越权访问 / 直接调 API

- **步骤**：以 finance 的 token 调 `POST /api/anchors`
- **预期**：返回 403

#### TC-SEC-07 未登录访问受保护接口

- **步骤**：不带 token 调 `GET /api/dashboard/overview`
- **预期**：401

#### TC-SEC-08 重复账号 / 重复身份证

- **步骤**：尝试创建同名用户、同身份证主播
- **预期**：返回 400/409 提示重复（如未做唯一约束需补；这是已知风险点）
